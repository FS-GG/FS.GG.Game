# `template/` — FS.GG.Game producer content

The producer-side template content FS.GG.Game owns after the ADR-0022 extraction.

## `product-skills/`

The **twelve game product skills** this repo publishes, in three groups.

**ADR-0022 P4 — migrated byte-identically** (skill-ownership `fs-gg-rendering → fs-gg-game`):
`fs-gg-game-core`, `fs-gg-audio`, `fs-gg-persistence`, `fs-gg-model-swap`. Each `SKILL.md` began
byte-identical to the copy FS.GG.Rendering still ships from its **frozen**
`dotnet new fs-gg-ui --profile game` (the accepted two-copies cost — ADR-0022 §6).

**ADR-0022 P6 — migrated and rewritten** (FS.GG.Game#35): `fs-gg-collision`, `fs-gg-grids`,
`fs-gg-line-drawing`, `fs-gg-visibility`. The guidance follows the implementation: #32/#33/#34/#38
pulled the backing code down into `FS.GG.Game.Core` as `Los`/`Fov`/`Visibility`/`Grids`/`Resolution`.
These are **not** byte-identical to Rendering's copies and cannot be — Rendering's bodies teach a
product-owned `Collision.fs`/`Visibility.fs`/`Grids.fs`/`LineDrawing.fs` fragment, while the
authoritative implementation here is a package module with a different surface. The divergence is the
`game-starter-two-copies` registry row, which is `coherent: false` for exactly this reason.

**Originating here** (never in Rendering): `fs-gg-ballistics`, `fs-gg-ai`, `fs-gg-effects`,
`fs-gg-physics`. Authored in this repo, so Rendering ships no copy of their `SKILL.md` and the
two-copies divergence above does not touch these **bodies** — which is a claim about the skills, not
about the code beneath them (`fs-gg-physics` teaches `Loop.advance`, whose accumulator does duplicate
one in FS.GG.UI.Canvas).

### Every pointer in these bodies must resolve

A published body makes three kinds of promise to its reader, and `scripts/check-skill-refs.sh`
(`gate.yml` → *Dangling skill refs*) checks all three, over every `.md` under `product-skills/`:

- **`[[wiki-ref]]`** — bare when it names a skill published here, otherwise qualified with the
  publishing repo's registry owner id (`[[fs-gg-rendering:fs-gg-scene]]`).
- **An issue/PR link** — must resolve *and be open*. The prose around one almost always tells the
  reader to go and do something at the other end, and a closed issue is a place nobody can act.
- **A bare `#N`** — an **error**, and the rule to know if you are authoring here. These bodies
  materialize into **somebody else's repository**, and GitHub resolves `#N` against the repo it is
  *read* in: in a scaffolded product it points at that product's own tracker, never back at ours. So
  its upstream state is beside the point, and repointing it cannot fix it — the fix is to qualify it
  (`FS.GG.Game#208`), which survives materialization.

The last two are strict by default, so each has a named escape; an honest exception costs one line
and is self-documenting to the next reader. Cite the issue that *implemented* something (history
stays closed), or mark a number that is prose rather than a pointer:

```
<!-- skill-refs: closed-ok FS.GG.Rendering#245 — cited as the issue that wired the seam -->
<!-- skill-refs: prose-ok #1 — the design doc's number-one bug, not issue #1 -->
```

The header of `scripts/check-skill-refs.sh` carries the full convention and the reasoning for each
rule; it is the source of truth this section summarizes.

## `skill-manifest/skill-manifest.json`

The **producer skill-manifest** (schema v1) declaring those twelve skills — the contract the org
registry `registry/skills.yml` (FS-GG/.github) reconciles its `owner: fs-gg-game` rows from, so
**registry = manifest = bytes**. Generated by `scripts/generate-skill-manifest.fsx` (recomputes each
`SKILL.md` digest); a CI gate (`gate.yml` → *Skill-manifest drift*) fails on any drift.
`materializes-when` is written in the ADR-0017 **canonical** grammar the skill-union gate evaluates.

## Not here yet

The `dotnet new fs-gg-game` **template package**, its `.template.config/template.json`, the game
**fragments** (`template/fragments/{collision,grids,line-drawing,visibility}` — still frozen in
Rendering; #35 moved the *skills*, not the emitted source), and the consumer **scaffold provider**
are deferred to the sequel provider epic (ADR-0022 §2.1 "later" / P6). This directory is content +
manifest only; nothing here is wired to a scaffold yet.
